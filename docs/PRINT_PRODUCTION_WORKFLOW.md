# Print Production Workflow — Architecture Spec

**Status:** Draft for review, 2026-05-25
**Owner:** Pavel + Claude
**Scope:** End-to-end lifecycle from creator's "Pay" button to finished goods leaving the production floor. Covers file validation, prepress review, approvals, pricing corrections, production release, accountability, and exception handling.
**Not in scope:** FDA / regulatory compliance checks (that's a separate gate, handled by the existing `services/compliance/` Python service). Print preflight answers *"can this file be printed correctly?"* — compliance answers *"is this label legal to sell?"* Both are independent gates; both must pass before production.

---

## TL;DR for executives

Five layers of validation, in order of leverage:

1. **Prevent at design time** — the canvas constrains creators so most issues are mathematically impossible. *Highest leverage. Build this carefully.*
2. **Automated preflight on submission** — partner-specific PDF preflight catches 70-80% of remaining issues in seconds.
3. **Risk-triaged human review** — only medium / high-risk jobs touch a human. Low-risk jobs auto-release.
4. **Explicit approval gates** — creator approves the proof (accepts design liability), provider releases to production (accepts execution liability). Liability transfer is the legal spine of the system.
5. **Change orders for mid-stream changes** — if reality diverges from the approved proof, a formal Change Order re-quotes and re-approves. Prevents disputes.

The single biggest mistake platforms make is allowing free-form chat between creator and provider as the dispute-resolution mechanism. Replace it with structured Revision Requests + a hard cap on revision rounds + automatic escalation to platform ops. This is the difference between scaling to 1,000 orders/day and drowning in support tickets.

## 1. The five validation layers

### Layer 1: Prevent at design time (highest leverage)

> *The cheapest preflight error is the one the creator could never create in the first place.*

Constraints we bake into the Design Studio canvas:

| Constraint | How it's enforced | Industry precedent |
|---|---|---|
| Bleed / safe-area / trim geometry | Die-cut frame on canvas physically prevents object placement outside zones | Vistaprint, Packlane |
| Color gamut | Color picker shows only colors in the *active partner's* supported gamut (Pantone library for offset; CMYK only for digital; brand swatches always allowed but flagged if out-of-gamut) | Cimpress prepress profiles |
| Resolution | Image placed on canvas → DPI calculated against final print size → warning chip on the image if below the partner's minimum | Stickermule, Printful |
| Fonts | Catalog-only fonts (per Q3 decision). Platform owns the font files and outlines them on export. Creators *cannot* introduce a missing-font failure. | Canva, Adobe Express |
| Barcodes / QR codes | Generated by platform from validated input data (GTIN/UPC/EAN/Code-128/QR). Validation happens at entry, not at print time. | All major printers |
| Dimensions / die-line | Encoded in the chosen DieCutTemplate. Creator cannot change the canvas dimensions. | Packlane (extreme version) |
| Transparency / overprint | Canvas serializes to Fabric.js JSON. Export pipeline flattens to PDF/X-4 deterministically. Creator can't ship broken transparency. | Adobe PDF/X standard |
| File format | Output is always PDF/X-4 (or partner-preferred variant like PDF/X-1a). Creator never picks the format. | ISO 15930 |

**Investment ratio:** spend 10× on Layer 1 polish for every 1× you spend on Layers 2-5 patching for things Layer 1 should have prevented. Packlane is the gold standard here — their canvas is so constrained that their preflight failure rate is reportedly under 2%.

### Layer 2: Automated preflight on submission

Triggered by "Next →" from the canvas. Runs server-side in a queue worker (a new service: `services/preflight/`).

**Input:** Canvas state (Fabric JSON) + chosen partner + chosen materials.
**Process:**

1. **Render** canvas to PDF/X-4 using a headless renderer (recommended: `pdfkit` + `harfbuzz` for typography, or `Ghostscript` for the PDF/X envelope).
2. **Preflight** the PDF against the **partner-specific PreflightProfile** (see §8).
3. **Generate** a `PreflightReport` with one of three outcomes: `PASS`, `WARN`, `FAIL`. Reports include per-issue items with severity, location (page + bounding box), human-readable explanation, and suggested fix.
4. **Persist** the PDF as a `ProductionFile` row with `version=1`.

**Outcomes:**

- `PASS` → advance state to `PREFLIGHT_PASSED`. Continue to risk-triage (Layer 3).
- `WARN` (e.g. image is 280 DPI when 300 recommended) → creator sees a *Review Warnings* modal. They can `Acknowledge and Continue` (warning is logged with their explicit accept) or `Return to Canvas` to fix.
- `FAIL` (e.g. text outside safe area, color out of gamut) → creator must fix. Cannot proceed.

**Industry tooling we should NOT rewrite from scratch:**

- **Enfocus PitStop Server** — the industry-standard PDF preflight engine. Sold to print houses as a server license. Profiles are XML; can be authored in PitStop Pro. Expensive (~$2-5K/yr per seat) but battle-tested.
- **callas pdfToolbox** — Enfocus's main competitor. Cheaper. Programmable via CLI / SDK.
- **Ghostscript + custom rules** — DIY option. Cheaper but needs significant rule-authoring work.
- **VeraPDF** — open-source PDF/A validator. Useful for PDF/A but limited for the full PDF/X preflight scope.

**Decision (Pavel 2026-05-25): Enfocus PitStop Server for V1.** The strategic framing is that we are building a multi-party manufacturing marketplace, not a print-preflight company. The hidden risk of DIY preflight is not software cost — it's that a single high-profile miss (overprint, spot color, trapping, transparency flattening, embedded profile mismatch, knockout behavior) costs provider trust permanently. PitStop has industry legitimacy + mature PDF/X handling + profile portability + provider familiarity. That reduces onboarding friction enormously.

**Engine-agnostic abstraction layer.** To avoid vendor lock-in, the preflight engine is abstracted behind an interface from day one:

```ts
interface PreflightEngine {
  run(file: ProductionFile, profile: PreflightProfile): Promise<PreflightResult>
}

class PitStopServerEngine implements PreflightEngine { ... }   // V1 implementation
class GhostscriptCustomEngine implements PreflightEngine { ... } // V2+ alternative
```

V2 re-evaluation triggers (any of):
- 10k+ orders processed → real error analytics available
- Known dominant issue patterns we can target with custom rules
- PitStop pricing crosses a margin threshold
- Customer-segment-specific preflight needs PitStop can't express

### Layer 3: Risk-triaged human review

Not every job needs a prepress technician. Triage rule (the **Risk Score**):

```
risk_score = 0
risk_score += 10  if print_process IN ('offset', 'flexo')   # higher-stakes processes
risk_score += 5   if finishing IN ('foil', 'embossing', 'spot_uv', 'die_cut_custom')
risk_score += 5   if has_pantone_spot_colors
risk_score += 5   if MOQ >= 1000
risk_score += 3   if order_total_cents >= 100_000          # $1000+
risk_score += 5   if creator.order_count < 3               # new creator
risk_score += 3   if preflight_warnings_acknowledged > 0   # creator overrode warnings
risk_score -= 5   if creator.successful_orders_with_this_partner >= 5

if risk_score <= PARTNER.autoReleaseMaxRiskScore:  route = AUTO_RELEASE
elif risk_score <= 15:                              route = PROVIDER_QUICK_REVIEW
else:                                               route = PROVIDER_FULL_PREPRESS
```

**Launch threshold (Pavel 2026-05-25): `autoReleaseMaxRiskScore = 3`, NOT 5.** Conservative-first. The first operational priority is learning failure modes — NOT maximizing automation volume. At launch, creators are inexperienced, providers distrust automation, profiles are immature, and edge cases are unknown. Optimizing for automation before understanding the error distribution is the most common early-stage mistake in production platforms.

**Phased rollout strategy:**

- **Phase 1 (months 0-3):** `autoReleaseMaxRiskScore = 3` platform-wide. Goal: gather statistics safely. Every auto-released order is tracked for downstream dispute / revision / quality outcomes.
- **Phase 2 (after low dispute + low revision + low provider-override rates prove safety):** expand selectively to `≤5`. Per-partner basis first, then platform default.
- **Phase 3 (post-Phase 2 stability):** allow individual partners to push to ≤7 if their record is exceptional.

**Per-partner override.** Each `PartnerService` carries `autoReleaseMaxRiskScore` (default = global value). Aggressive-automation partners can opt up; conservative partners can lower (even to 0 = "touch every file"). This flexibility increases partner adoption — enterprise print houses and small digital shops have very different risk tolerances; forcing them into one model creates friction. (See §10 for Provider Confidence Modes which automate this per-partner tuning.)

The numbers above are *initial defaults* — they get tuned with data over time. The Risk Score is logged on every order so we can audit "what would have happened under different rules" retrospectively. **Always log the full score breakdown** (which factors fired with what weight) so admin can analyze in aggregate.

**Routes:**

| Route | Who reviews | What they do | SLA |
|---|---|---|---|
| `AUTO_RELEASE` | Nobody | Order auto-advances to PROOF_PENDING_CREATOR (creator still must approve their own proof) | Immediate |
| `PROVIDER_QUICK_REVIEW` | Provider Prepress Tech | ≤5 min spot-check of the auto-passed file. Look for obvious issues the preflight missed (visual judgment: ugly trapping, awkward color, weird scaling). Approve / reject with reason / request creator revision. | 4 business hours |
| `PROVIDER_FULL_PREPRESS` | Provider Prepress Tech | Full review. May generate a soft proof (PDF) or hard proof (mailed physical sample). May call creator to discuss. May request revisions. | 24 business hours (soft proof) / 5 business days (hard proof) |

Auto-release path is essential to keep small/low-stakes jobs (50 stickers, business cards, 100-unit digital print run) economically viable. Without it, providers' prepress queues clog with trivial work and creators wait days for things that should ship same-day.

### Layer 4: Approval gates — explicit liability transfer

This is the legal spine. Each gate is a **separate audit-logged event** with timestamp, actor (creator user OR provider tech user), the exact file hash reviewed, and a click-through accepting the terms of approval.

**Gate A — Creator Approves Proof:**

The creator sees the final press-ready PDF (or hard proof image) and a checkbox modal:

> "I approve this design for production. I confirm:
> - All text is spelled correctly
> - All required information appears as I intended
> - I accept that the printed product will match this exact file
> - I accept that revisions after this point require a Change Order and may incur additional cost
>
> ☐ I approve. Release to production →"

The act of clicking transfers **design liability** to the creator. If the printed product matches the approved file but the creator notices a typo two weeks later, that's the creator's problem.

**Gate B — Provider Releases to Production (one per involved partner, parallel):**

A production order typically touches multiple partner services simultaneously: a manufacturer (co-packer), a label printer, and possibly a warehouse (when shipping to a connected WAREHOUSE service). **Each involved partner must independently pass Gate B for their slice of the order.** Reviews happen in **parallel**, not serially — each partner sees the order at the same time and can approve / reject / request revision on their portion independently.

Per-partner Gate B click-through:

> "I confirm the file I am loading into production matches hash `[SHA-256]`, which was approved by the creator at `[timestamp]`. I accept responsibility for the [printed label / packaging fill / fulfillment handling] matching this file within our published tolerances (Delta-E ≤ 4 for solid colors, ≤ 6 for photographic). [Partner name] accepts execution liability."

The act of clicking transfers **execution liability for that partner's slice** to that partner. The order does NOT transition to `PRODUCTION_LOCKED` until ALL involved partners have completed their Gate B.

**Payment-capture rule (Pavel 2026-05-25):** Stripe holds the customer's authorization but does NOT capture until ALL involved partners have completed Gate B AND the creator has completed Gate A. Capture happens at the moment the order enters `PRODUCTION_LOCKED`. If any partner rejects hard, the order moves to `ORDER_HOLD` and the authorization either gets voided (if order canceled) or held longer (if rerouted to a backup partner). See §5 for the full capture flow + authorization-expiry handling.

**Why parallel and not serial:** sequential review wastes 2-3× the elapsed time and provides no value — each partner reviews their own slice independently, not the others'. Parallel review is industry standard in packaging/print supply chains.

**Why this matters:** without explicit per-partner gates, every dispute devolves into "you should have caught it" vs "you should have told me." With one gate per partner, dispute resolution is mechanical: *whose gate was violated?* And payment incentive aligns with response speed — partners only get paid when everyone has signed off.

### Layer 5: Change orders for mid-stream changes

Once `PRODUCTION_LOCKED`, the file is frozen. Any change requires a formal **Change Order**:

```
ChangeOrder {
  id
  orderId
  initiatedByUserId               # creator OR provider
  reason: enum                    # CREATOR_REQUESTED_EDIT | PROVIDER_FOUND_ISSUE
                                  # | SPEC_CHANGE | SUBSTRATE_UNAVAILABLE | OTHER
  description: string
  delta:                          # what's changing
    fileChanges: ProductionFile? # null if only spec / metadata changes
    quantityDelta: int?
    substrateDelta: string?
    finishingDelta: string?
    addressDelta: jsonb?
  priceDelta: int                 # cents; positive = creator pays more
  status: PENDING_QUOTE | PENDING_APPROVAL | APPROVED | REJECTED | EXPIRED
  ...
}
```

Flow:

1. Either party initiates a Change Order.
2. Provider re-quotes (price delta) within 1 business day SLA.
3. Creator approves or rejects within 3 business days SLA.
4. If approved → state returns to `PROOF_PENDING_CREATOR` for re-approval of the new file/spec; then back to `PRODUCTION_LOCKED`.
5. If rejected → original state restored. Order continues with original spec.

**Industry norm:** print houses treat change orders as legal documents. Cimpress / Tag / HH Global all use them. Avoiding them in favor of "let's just chat about it" is how platforms hemorrhage money on disputes.

## 2. Roles

A new persona joins the existing 5-layer partner onboarding model: **Provider Prepress Technician**.

| Role | Existing? | What they do |
|---|---|---|
| **Creator** | Yes | Designs in canvas, approves proofs, accepts design liability, raises change orders |
| **Platform Preflight Service** | New | Automated. Runs PreflightProfiles, generates PreflightReports, makes the risk-triage decision. No human in the loop. |
| **Provider Prepress Technician** | **NEW persona** | Reviews flagged files. Approves / rejects / requests revisions. Generates proofs. *Belongs to the Provider organization, not iLaunchify.* |
| **Provider Production Operator** | New | Loads the locked file into production. Accepts execution liability via Gate B. |
| **Platform Operations** | Yes (admin) | Escalation only. Mediates disputes. Reassigns orders to backup providers. Cannot make production decisions, but can issue refunds and refund-on-redo orders. |

**Schema implication:** the Partner User model needs role types. The current schema has `Partner.userId @unique` — meaning today one Partner row = exactly one User account. **That constraint has to go** to support multi-user partner teams. We need a `PartnerMembership` junction model + invite flow. Details in §2.1-2.4 below.

The three roles + their permissions:

- `PARTNER_ADMIN` — full company control. Manages company profile, billing, payouts, team members + their roles, certifications, packaging catalog, pricing. Can also do everything PARTNER_PREPRESS and PARTNER_PRODUCTION can do (small shops won't bother creating separate users).
- `PARTNER_PREPRESS` — file review surface. Can view all incoming files for their partner's services, approve/reject/request revision, generate proofs, post Tier-2 comments. **Cannot** load files into production. **Cannot** change company info, team, or pricing.
- `PARTNER_PRODUCTION` — production surface only. Can view files only after `PRODUCTION_LOCKED`. Performs Gate B. **Cannot** modify files, change the spec, view pre-locked files, or change team / pricing / company info.

This separation is industry standard and prevents accidental "I changed the file at the press" disasters. Per [[ilaunchify-operational-philosophy-v1]]: keep the roles distinct in the schema even when small providers assign all three to one person.

### 2.2 Schema changes required

Important reality (Pavel 2026-05-25): a single Partner organization can offer **multiple services simultaneously** (`MANUFACTURING`, `COPACKING`, `LABEL_PRINTING`, `WAREHOUSE` — the existing ServiceType enum). Different humans typically handle each. A label-printing prepress technician shouldn't see the manufacturing recipe queue, and vice versa.

So roles split into **two scopes**:

- **Org-wide scope: `PARTNER_ADMIN`** (one row per user per Partner). Manages company profile, billing, payouts, team, certifications. Admin always sees everything across all of the Partner's services.
- **Service-scoped: `PARTNER_PREPRESS` / `PARTNER_PRODUCTION`** (one row per user per PartnerService). Each row ties a specific human to a specific service. A polymath who works on both label printing and manufacturing has two rows.

```prisma
// REMOVE the @unique constraint from Partner.userId.
// The userId column itself can stay as "founder of record" pointer.

// Org-wide membership: who belongs to the Partner organization at all.
// Carries org-wide admin flag and shared per-user metadata (2FA, last-active).
model PartnerMembership {
  id              String                @id @default(cuid())
  partnerId       String
  userId          String
  isAdmin         Boolean               @default(false)     // org-wide PARTNER_ADMIN
  invitedByUserId String?                                  // null only for the founder
  invitedAt       DateTime?
  acceptedAt      DateTime              @default(now())
  removedAt       DateTime?                                 // soft-remove; audit log preserved
  removedByUserId String?
  removedReason   String?
  lastActiveAt    DateTime?
  twoFactorEnrolled Boolean             @default(false)
  partner         Partner               @relation(fields: [partnerId], references: [id], onDelete: Cascade)
  user            User                  @relation(fields: [userId], references: [id], onDelete: Cascade)
  serviceMemberships PartnerServiceMembership[]
  @@unique([partnerId, userId])
  @@index([partnerId, removedAt])
}

// Service-scoped memberships: which work roles this user holds on which
// specific PartnerService (MANUFACTURING / COPACKING / LABEL_PRINTING / WAREHOUSE).
// One user can hold rows for multiple services (label printer + recipe reviewer).
// One service can have multiple users (lead + associate, see §2.7).
model PartnerServiceMembership {
  id                  String              @id @default(cuid())
  partnerMembershipId String                                // FK into the org-wide row
  partnerServiceId    String                                // FK to specific PartnerService
  roles               PartnerServiceRole[]                  // [PARTNER_PREPRESS] | [PARTNER_PRODUCTION] | both
  removedAt           DateTime?
  removedByUserId     String?
  removedReason       String?
  createdAt           DateTime            @default(now())
  partnerMembership   PartnerMembership   @relation(fields: [partnerMembershipId], references: [id], onDelete: Cascade)
  partnerService      PartnerService      @relation(fields: [partnerServiceId], references: [id], onDelete: Cascade)
  @@unique([partnerMembershipId, partnerServiceId])
  @@index([partnerServiceId, removedAt])
}

// Invite carries both scopes — what to grant on accept.
model PartnerInvite {
  id              String                @id @default(cuid())
  partnerId       String
  invitedByUserId String                                    // must be PARTNER_ADMIN at invite time
  emailAddress    String
  grantAdmin      Boolean               @default(false)     // org-wide admin on accept?
  serviceGrants   Json                                      // [{serviceId, roles: [PREPRESS, PRODUCTION]}]
  token           String                @unique
  tokenExpiresAt  DateTime                                  // 14 days
  status          PartnerInviteStatus   @default(PENDING)
  consumedAt      DateTime?
  consumedByUserId String?
  revokedAt       DateTime?
  revokedByUserId String?
  createdAt       DateTime              @default(now())
  partner         Partner               @relation(fields: [partnerId], references: [id], onDelete: Cascade)
  @@index([emailAddress, status])
  @@index([partnerId, status])
}

enum PartnerServiceRole {
  PARTNER_PREPRESS
  PARTNER_PRODUCTION
}

enum PartnerInviteStatus {
  PENDING
  CONSUMED
  REVOKED
  EXPIRED
}
```

**Why split into two tables instead of a single `PartnerMembership.roles[]` enum:**

- Roles need to be scoped to a specific service, not just a "type" of work. Mike-the-label-prepress-tech should not see Linda's recipe-review queue even though both are "prepress" in the abstract.
- A user often holds *different* roles on different services (admin org-wide + prepress on label printing + production on warehouse). Flat arrays of `[PARTNER_ADMIN, PARTNER_PREPRESS, PARTNER_PRODUCTION]` lose the per-service distinction.
- When a Partner adds or removes a service (e.g. adds WAREHOUSE in V1.5), we only need to add/remove PartnerServiceMembership rows — the org membership is untouched. Clean cascade.
- Query patterns are clean: "all members of Acme who do label-printing prepress" = `PartnerServiceMembership WHERE partnerServiceId = X AND roles CONTAINS PARTNER_PREPRESS AND removedAt IS NULL`.

**The original enum `PartnerUserRole` from earlier draft is split:** `PARTNER_ADMIN` becomes a boolean (`PartnerMembership.isAdmin`) because it's org-wide and a user either has it or doesn't. `PARTNER_PREPRESS` and `PARTNER_PRODUCTION` become the service-scoped `PartnerServiceRole` enum.

### 2.4 Edge cases + safeguards

- **Last-admin lockout protection:** the last PARTNER_ADMIN cannot remove their own admin flag or remove themselves entirely until another PARTNER_ADMIN exists on the team. Surface a "promote someone first" inline error.
- **Last-service-member protection (see also §2.7):** removing the last PARTNER_PREPRESS or PARTNER_PRODUCTION on a service triggers a soft warning, not a hard block — admin chooses.
- **Role change vs revoke:** changing a user's roles (org-wide or per-service) is a low-stakes admin action. Removing them entirely is high-stakes (their queue items get reassigned, audit entries marked archived-user). Both write to AuditLog. Removal requires a confirm modal with the reason.
- **Soft-remove, never hard-delete:** `removedAt` flags rows as inactive on both `PartnerMembership` and `PartnerServiceMembership`. The User account itself isn't touched (they may belong to other Partners or have other roles).
- **Same user, multiple Partners (V1.5+):** schema supports it via the junction model. V1 UI assumes one Partner-at-a-time. If a user is somehow attached to two Partners, V1 picks the most-recently-active and shows a "switch partner" link. V1.5 builds a proper partner-switcher.
- **2FA requirements:**
  - **PARTNER_ADMIN (org-wide):** mandatory. Controls billing + payouts + team — biggest blast radius if compromised.
  - **PARTNER_PRODUCTION (any service):** strongly recommended. Presses the Gate B button = legal liability transfer for that production run.
  - **PARTNER_PREPRESS (any service):** optional but recommended.
  - Enforce mandatory 2FA at first login after the role is assigned; provide TOTP setup wizard.
- **Pending invites:** PARTNER_ADMIN can see + revoke + resend pending invites from /partner/team. Revoking invalidates the token immediately.
- **Email change:** if a teammate updates their email, the membership remains tied to the User id (not email), so nothing breaks. Audit log captures the email change.
- **Invite to a non-ACTIVE Partner:** blocked at the action level. The /partner/team page itself isn't shown until the partner reaches ACTIVE state.

### 2.5 What this hooks into in the existing codebase

- `apps/partner/src/app/signup/` — unchanged. Still creates one User + one Partner + (NEW) one `PartnerMembership(isAdmin=true)` for the founder.
- `apps/partner/src/app/(dashboard)/team/` — **new directory**: `page.tsx` (list + invite button), `actions.ts` (invite / revoke / re-role server actions), `InviteModal.tsx` (with the per-service role picker from §2.7).
- `apps/partner/src/app/invite/[token]/page.tsx` — **new**: invite acceptance landing (signup-or-login).
- Auth middleware (`packages/auth/`) — must consume `PartnerMembership` + `PartnerServiceMembership` instead of `Partner.userId` for the "which partner is this user acting as? which services can they see?" lookup. Provides three small helpers: `requirePartnerAdmin()`, `requireServiceRole(partnerServiceId, role)`, `currentUserServiceIds()`.
- All existing `partner.creatorProfile.userId === user.id`-style ownership checks (used in creator-side actions) are unaffected because Creator hasn't gained team support.
- Notification dispatcher (`@ilaunchify/notifications`) — gain two new routing primitives: "to all admins of partner X" and "to all members of partner X with role Y on PartnerService Z". The second one is what powers "new label-printing order arrived → notify only the LABEL_PRINTING prepress techs."
- Order queues already keyed by `PartnerService.id` — filtering by "services this user belongs to" via PartnerServiceMembership becomes the natural query.

### 2.6 Why this is gated to post-ACTIVE

Allowing team management during onboarding would let an unverified company create multiple accounts that could view sensitive admin-side reviews of their own application. Worse: it creates an avenue for fraud (one person signs up for verification, then quietly hands off to a different person who passes Gate B for real production orders).

Holding team management until ACTIVE keeps the verification chain accountable to a single identifiable founder. Once verified, the company can scale its team safely.

### 2.7 Multi-service partners + associates (Pavel 2026-05-25)

This section answers two direct questions:
1. *"Partner role is also Manufacturer / Print provider / Co-packer / Warehouse — and might be a combination of a few. How does this work?"*
2. *"They might need for each major role an associate (another user who needs to be invited and work on the processes). How is this decided?"*

#### Multi-service partners

The existing schema already supports this — a Partner has many `PartnerService` rows, each with its own `type` (`MANUFACTURING` / `COPACKING` / `LABEL_PRINTING` / `WAREHOUSE`). The team model layered on top:

```
Acme Industries (Partner)
├── PartnerService: LABEL_PRINTING       ┐
│      ├── Mike (PARTNER_PREPRESS)       │  separate humans per service
│      └── Tom (PARTNER_PRODUCTION)      │
├── PartnerService: MANUFACTURING         │
│      ├── Linda (PARTNER_PREPRESS)      │
│      └── Carlos (PARTNER_PRODUCTION)   ┘
└── PartnerService: WAREHOUSE
       └── Jenny (PARTNER_PRODUCTION = the warehouse intake operator)

Sarah is org-wide PARTNER_ADMIN — sees all three queues + manages everyone.
```

**Order routing example:** a creator's order comes in with `manufacturerServiceId = Acme.MFG`, `printProviderServiceId = Acme.LABEL`, `shipToPartnerServiceId = Acme.WAREHOUSE` (all three at the same Partner). The platform creates three `OrderPartnerApproval` rows (one per PartnerService). Linda + Carlos see only their slice in MANUFACTURING. Mike + Tom see only their slice in LABEL_PRINTING. Jenny sees only her slice in WAREHOUSE. Sarah sees all three.

**Polymath case:** if Linda also handles label prepress because the team is small, she gets a second `PartnerServiceMembership` row for `LABEL_PRINTING`. Now she sees both queues. No special-case code; same junction model.

**Service-type-specific UI:** each queue page is parameterized by service type so the UI can render the right widgets:
- `LABEL_PRINTING` prepress queue → preflight report, soft-proof viewer, file annotation tools
- `MANUFACTURING` prepress queue → recipe review, allergen audit, certification check
- `WAREHOUSE` queue (no real "prepress") → just the production-release Gate B for inbound shipment intake
- `COPACKING` → similar to MANUFACTURING but with packaging-system focus

These different prepress experiences are why service-scoping matters in the schema, not just in the UI.

#### Associates (the second question)

"Associate" means: when Mike (lead prepress tech for labels) wants to bring Jenny on as his backup / assistant. They share the queue; orders can be picked up by either.

**Three possible models — recommendation is Option 1 for V1:**

| Option | How | Pros | Cons |
|---|---|---|---|
| **1. Multiple equal memberships** *(recommend V1)* | Just invite Jenny with the same `PARTNER_PREPRESS` role on the same `LABEL_PRINTING` PartnerService. She and Mike are equal in the system; the partner manages seniority internally. | Simplest schema. Matches how Stripe / Linear / Notion handle team hierarchy. Most providers don't need formal seniority. | No platform-visible distinction between lead and associate. |
| **2. Seniority field** *(V1.5 if real demand)* | Add `PartnerServiceMembership.seniority: LEAD \| PRIMARY \| ASSOCIATE`. Affects nothing in V1 except notification routing (LEAD gets the first notification; ASSOCIATE gets one only if LEAD doesn't respond within X hours). | Lets us route alerts cleanly when partners have explicit hierarchy. | Premature optimization — most small teams don't need it; large teams have their own internal queues. |
| **3. Role variants in the enum** | Add `LEAD_PREPRESS` / `SECONDARY_PREPRESS` to `PartnerServiceRole`. | Most explicit. | Pollutes the role enum with seniority concepts; doesn't compose well when a person is "lead on labels but associate on mfg." |

**Final recommendation for V1:** Option 1. Multiple equal memberships per service. Three reasons:

1. **It matches B2B norms.** Stripe, Linear, Notion, Figma, GitHub Organizations — none of them ship lead/associate distinctions in their core role model. Teams self-organize.
2. **Partners can simulate seniority via notification preferences.** Mike sets quiet-hours OFF; Jenny sets quiet-hours ON for night-time. Mike picks up urgent orders; Jenny handles overflow.
3. **We can add Option 2 in V1.5 without breaking V1.** The schema migration would be: add a nullable `seniority` column with default = NULL = no opinion. Existing rows stay equivalent.

If you push back and want explicit lead/associate from day one, I'd implement Option 2 with `seniority` defaulting to `PRIMARY` for the first member added to a service, and `ASSOCIATE` for everyone subsequent. Easy to flip the default.

#### Invite modal UX implication

The invite form needs to handle the multi-service + multi-role case cleanly:

```
┌──────────────────────────────────────────────────┐
│ Invite teammate to Acme Industries               │
├──────────────────────────────────────────────────┤
│ Email:    [_________________________________]    │
│                                                  │
│ Organization-wide:                               │
│   ☐ Make this person an admin                    │
│                                                  │
│ Service access — pick role(s) per service:       │
│                                                  │
│   📦 Label Printing                              │
│      ☐ Prepress technician                       │
│      ☐ Production operator                       │
│                                                  │
│   🏭 Manufacturing                                │
│      ☐ Prepress technician                       │
│      ☐ Production operator                       │
│                                                  │
│   📋 Co-packing                                  │
│      ☐ Prepress technician                       │
│      ☐ Production operator                       │
│                                                  │

```

The modal only lists services the Partner has actually activated. If Acme only does LABEL_PRINTING + MANUFACTURING, the COPACKING and WAREHOUSE sections don't render. Admins can grant org-wide admin alone (no service access) for someone like a CFO who needs to see billing but no production work.

#### Edge cases specific to multi-service

- **Adding a new service post-team-setup:** Acme already has LABEL_PRINTING with 4 team members. They get approved for MANUFACTURING. The new service has zero memberships — dashboard surfaces an alert: "Manufacturing has no team members yet. Invite someone or assign existing teammates."
- **Removing a service:** Acme pauses their MANUFACTURING service. All `PartnerServiceMembership` rows scoped to that service auto-soft-remove (set `removedAt`). When they reactivate, members can be restored or re-invited.
- **Org admin without any service access:** allowed. A CFO might have org-wide admin (billing, payouts) but no production work. The dashboard renders their queue-less view appropriately.
- **Last service-member protection:** if Mike is the only `PARTNER_PREPRESS` on `LABEL_PRINTING` and tries to leave (or admin tries to remove him), the system warns: "This service will have zero prepress technicians — incoming label orders will sit in queue unassigned. Continue?" Soft-warn, not hard-block — admin can choose to assume the role temporarily.

#### Why we didn't fold this into the original team model

The simple model (§2.1-2.6) is the right *mental* model — humans onboard the company first, then admins invite teammates. The multi-service nuance is real but it's a **data-shape** consideration, not a fundamental flow change. The invite flow is identical; only the invite modal carries an extra dimension. This is why the rest of §2 stays intact and only §2.2 (schema) and this §2.7 (multi-service mechanics) needed extension.

## 4. Communication protocol

The communication layer has **two tiers**, working together but strictly separated. This is a refinement (Pavel 2026-05-25) of the original "no free-form chat" position — that was too purist.

**Tier 1 — Structured Revision Requests (drive the workflow).**
Tier 2 — Per-Order Comment Thread (carries context, never drives state).

### 4.1 Tier 1: Structured Revision Requests

Handle ~80% of real cases. Every preflight-style issue uses a fixed vocabulary so reports are consistent across providers and disputes are mechanically resolvable.

```
RevisionRequest {
  id
  orderId
  round: int                          # 1, 2, 3 max
  raisedByUserId                      # provider tech user
  raisedAt: timestamp
  items: RevisionItem[]
}

RevisionItem {
  id
  revisionRequestId
  issueCode: enum                     # see vocabulary below
  severity: BLOCKING | WARNING
  location:                           # where in the file
    pageNumber: int
    boundingBoxPdfPoints: jsonb?      # x, y, w, h
    objectId: string?                 # the Fabric.js object id if known
  description: string                 # max 280 chars (Twitter rule)
  suggestedFix: enum                  # see vocabulary
  exampleImageAssetId: string?        # provider can attach a screenshot
}
```

**Shared issue vocabulary** (extensible, but always from this enum — no free-text "issue type"):

| Issue code | Plain English | Typical suggested fix |
|---|---|---|
| `LOW_RESOLUTION` | Image is below DPI threshold | "Upload a higher-resolution version" |
| `OUT_OF_GAMUT_COLOR` | Color won't print accurately | "Pick from the partner's gamut" |
| `BLEED_INSUFFICIENT` | Object too close to trim | "Extend background to bleed line" |
| `SAFE_AREA_VIOLATION` | Text/logo too close to trim | "Move at least 3mm inside safe area" |
| `MISSING_BLEED` | Background doesn't extend to bleed | "Add bleed extension" |
| `RGB_DETECTED` | RGB image instead of CMYK | "Replace with CMYK version" |
| `FONT_NOT_OUTLINED` | (Only triggered if non-catalog font; shouldn't happen given Q3) | "Use only catalog fonts" |
| `TRANSPARENCY_FLATTENING` | Transparency may not print as expected | "Provider will flatten; review preview" |
| `BARCODE_TOO_SMALL` | Below scannable threshold | "Increase barcode size to ≥12mm" |
| `BARCODE_QUIET_ZONE` | Insufficient white space around barcode | "Add quiet zone" |
| `TRIM_MISMATCH` | File trim doesn't match die-line | "Re-import die-cut template" |
| `PANTONE_NOT_SUPPORTED` | Pantone color on a digital-only press | "Convert to CMYK or change partner" |
| `FOIL_LAYER_INVALID` | Foil/varnish spot layer naming wrong | "Use spot layer `iLaunchify_Foil`" |
| `MOQ_NOT_MET` | Quantity below partner's MOQ | "Increase quantity to ≥ X" |
| `WRONG_DIMENSIONS` | File size doesn't match selected SKU | "Resize to X × Y mm" |
| `ASPECT_DISTORTION` | Image stretched non-proportionally | "Re-place image at original ratio" |
| `OVERPRINT_BLACK_TEXT` | Black text missing overprint | "Set overprint=true on black text" |
| `OTHER` | Catch-all with description | "See description" |

The vocabulary is admin-editable and grows over time. But the act of forcing every issue into a code (with description filling in nuance) is what makes the system scalable and analyzable. After 10,000 orders we know exactly which issues dominate and can invest Layer 1 prevention in the right places.

**Hard rules for Tier 1:**

- Maximum 3 revision rounds per order. Round 4 auto-escalates to Platform Operations.
- Each round resets the SLA clock for the receiving party.
- Revision Requests are the only mechanism for design feedback that mutates order state.
- A revision can be **withdrawn** by the provider before the creator responds (e.g. "actually disregard, we'll handle it" — reverts to PROVIDER_REVIEWING).

**Escalating friction UX across the 3 rounds** (Pavel 2026-05-25). The cap matters psychologically — without friction signaling, creators don't internalize that rounds are scarce. The provider-side UI carries the same friction:

| Round | UI treatment | Effect on user behavior |
|---|---|---|
| **Round 1** | Standard. Quiet, low-stakes treatment. | Normal back-and-forth feels safe; encourages clean first submission. |
| **Round 2** | Warning banner: *"This is revision round 2 of 3. Try to address all concerns this round to avoid additional cycles."* | Creator feels the budget tightening. Provider gets prompted to batch all known issues into this round, not drip-feed. |
| **Round 3** | Red banner: *"Final revision round before automatic escalation to Platform Operations."* + a 1-click "Open Change Order instead" button + a "Talk to support" link. | Both sides decisively converge or escalate; no fourth round. |

After Round 3, the order auto-transitions to `ESCALATED_TO_OPS`. Platform ops can: reassign the slice to a backup partner, close-no-fault with full refund, or extend round count by 1 with documented justification (admin-only override).

**Abuse / risk metrics tracked per party:**

```
provider.revisionRoundAverage   = rolling 90-day mean rounds per APPROVED order
creator.revisionRoundAverage    = rolling 90-day mean rounds per APPROVED order
provider.revisionWithdrawalRate = fraction of raised revisions later withdrawn
creator.gateAReversalRate       = fraction of Gate A approvals followed by a Change Order within 24h
```

These become **trust/risk signals** feeding the risk score (§1 Layer 3) and partner-rating downstream. Examples of what they reveal:
- Provider averaging 2.8 rounds/job → either overly strict, poorly configured profile, or using revisions as unpaid consulting. Either way, detectable + addressable.
- Provider with high `revisionWithdrawalRate` → noisy reviewer, training opportunity.
- Creator with high `gateAReversalRate` → likely rushing approvals; UX should slow them down with extra confirmation.

### 4.2 Tier 2: Per-Order Comment Thread

For the ~20% of cases that don't fit a fixed vocabulary: creative consultation ("could be 4-color process or 3 spot colors, here are the trade-offs"), scheduling coordination ("can we push to Monday's run?"), edge-case judgment calls ("your asymmetric design — was that intentional?"), and trust-building on high-value or first-time partnerships.

The thread is a **scoped, time-bounded, audit-logged conversation pinned to a specific Order**.

```
OrderComment {
  id
  orderId
  authorUserId                       // creator, provider tech, or platform ops
  authorRole                         // CREATOR | PARTNER_PREPRESS | PARTNER_PRODUCTION | PLATFORM_OPS
  body                               // markdown, max 1000 chars
  createdAt                          // immutable; comments cannot be edited after 5 min
  editedAt?                          // within 5-min edit window only
  redactedAt?                        // platform ops can redact (not delete) abusive content
  redactedReason?
}
```

**Hard rules for Tier 2** (these are the ones that prevent the comment thread from devolving into chaos):

- **Cannot mutate order state.** Only Revision Requests, Change Orders, and approval gates can transition states. A comment that says "please change X" doesn't change anything — provider must raise a Revision Request.
- **Cannot override approval gates.** If creator approved Gate A and then comments "actually wait, change this," the approval still stands. To undo, creator must open a Change Order.
- **Cannot carry design files.** No file attachments in comments — would bypass preflight and create a shadow file system. Screenshots/annotations live inside Revision Requests where the structured workflow can audit them.
- **Rate-limited.** Max 20 comments per order across all parties. After 20, comment box disables with "Reached comment limit — open a Revision Request or Change Order to continue."
- **Time-bounded.** Thread closes automatically 30 days after order is `DELIVERED` or `CANCELED`. Read-only after close.
- **Visible to all parties + platform ops.** No DMs, no side channels. Creator sees provider's comments, provider sees creator's, platform ops sees everything.
- **No real-time presence / typing indicators.** This is async-by-design. Providers process orders in batches; the UI should not create an expectation of instant response.
- **Notification batching.** Comments are bundled into a daily digest email (not per-comment), unless the recipient explicitly opts into per-comment notifications.

**Explicit UI guidance** appears in/above the comment input every time:

> 💡 **For design feedback** (resolution, color, bleed, etc.), use **Request Revision**.
> 💰 **For spec changes** (quantity, substrate, finish), use **Change Order**.
> 💬 This thread is for context only — questions, scheduling, creative discussion.

### 4.3 Why two tiers instead of one (either way)

| Pure structured-only | Pure free-form chat | Two-tier (this design) |
|---|---|---|
| Covers ~80% of cases cleanly | Covers 100% of cases somehow | Covers 100% cleanly |
| Misses creative + contextual nuance | No audit structure | Structured workflow + contextual color |
| Forces high-value relationships into transactional mode | Scales O(messages) | Structured scales O(orders); contextual capped at 20 |
| Vistaprint / Stickermule / Printful | Etsy POD integrations / legacy white-label | Linear / Asana / Jira (closest analogs) |

Free-form chat alone failed at every print platform that tried it. Structured-only is fine for B2C transactional printing (Vistaprint / Moo) but underserves B2B production relationships where creative consultation matters. The two-tier model gives the workflow what it needs (structure) and the relationship what it needs (context).

## 5. Pricing + payment-capture model

**Pricing locks at `PRODUCTION_LOCKED`.** Before that point:

- Any change in spec (quantity, substrate, finish, etc.) auto-recalculates the quoted price live in the wizard's right-rail summary.
- The creator can see how each choice affects the total in real time.
- No payment is captured. At "Pay" click, Stripe only places an **authorization hold** (`payment_intent.confirm` with `capture_method='manual'`).

### 5.1 The capture rule (Pavel 2026-05-25)

**Stripe captures the payment only when ALL involved partners have approved the order via their Gate B + the creator has approved Gate A.** Not before.

Concretely, an Order can touch up to three PartnerServices:
- The **manufacturer** (`Order.manufacturerServiceId`) — fills/assembles the product
- The **label printer** (`Order.printProviderServiceId`) — produces the printed label
- The **warehouse** (`Order.shipToPartnerServiceId`) — receives finished goods (only when `shipToType=WAREHOUSE_PARTNER`)

Each gets its own approval row (`OrderPartnerApproval`, schema in §7). Each must reach status `APPROVED` independently. Reviews run in **parallel** — all involved partners see the order at the same time and decide on their own slice.

**Transition logic** (capture trigger):

```
on OrderPartnerApproval.update(status = APPROVED):
  involvedApprovals = all OrderPartnerApproval rows for this order
  if every row in involvedApprovals is APPROVED:
    if creator's Gate A is also APPROVED:
      order.allPartnersApprovedAt = now()
      order.status = PRODUCTION_LOCKED
      stripe.paymentIntents.capture(order.stripePaymentIntentId)
      order.paymentCapturedAt = now()
      order.lockedPriceCents = order.totalCents  # immutable from here
```

**Why this is better than capturing at submission:**

- Prevents the worst case where one partner captures payment and another rejects late, forcing an ugly multi-party refund
- Aligns financial incentive with provider SLA performance — fast review = fast capture = faster payout
- Provides clean rollback if any partner rejects hard: just void the authorization, no refund accounting
- Creator sees zero charge on their card statement until production is truly committed

### 5.2 Stripe authorization expiry handling

Card authorizations typically expire in **7 days** (longer for some processors / payment methods). If partner reviews drag on past the hold window, we must re-authorize before capture.

**Background worker** runs every 1 hour and:

1. Finds Orders in `AWAITING_PARTNER_APPROVALS` state where `paymentAuthExpiresAt` is within 24 hours
2. For each, attempts **off-session re-authorization** via Stripe (`payment_intent.confirm` re-run with the saved payment method)
3. On success: extends `paymentAuthExpiresAt` to the new expiry, notifies creator silently (in-app, no email noise)
4. On failure (e.g. card expired, insufficient funds): pauses the order, sends creator an "update your payment method" notification with a 72h grace before auto-cancellation

This is industry-standard behavior — same as how Airbnb / Booking.com / hotel reservations handle multi-day pre-auths.

### 5.3 Partner rejection handling

If any partner sets their `OrderPartnerApproval.status = REJECTED` (vs `CHANGES_REQUESTED`, which goes through the Revision Request flow), the order moves to `ORDER_HOLD`. Three resolution paths from there:

1. **Reroute** — platform ops or admin reassigns that slice to a backup PartnerService (V1.5+ feature; V1 requires manual ops intervention). The rejecting partner's `OrderPartnerApproval` row is closed; a new row opens for the backup. Other partners' approvals remain in force.
2. **Change Order** — creator agrees to a modified spec the partner can fulfill (e.g. switch from offset to digital, drop the foil accent). The CO triggers re-quote + creator re-approval + restart of partner approvals.
3. **Cancel + void auth** — full void of the Stripe authorization, order closed, no charge to the creator. No transfer to any partner since no money was captured.

### 5.4 Change Order pricing rules

- Provider must re-quote within 1 business day SLA.
- Quote includes the price delta and a one-paragraph rationale.
- If creator approves → Stripe charges (or refunds, if delta is negative) the difference *at the next capture* (not immediately — preserves the "no capture until all approvals" rule).
- If creator rejects → order continues at original spec/price.
- If provider initiates the Change Order due to their own error (e.g. they realized they quoted a substrate they don't carry) → they cannot charge more; only creator-initiated changes can result in upcharge.

### 5.5 Hard proof policy (Pavel 2026-05-25)

Hard proofs (physical printed samples mailed to the creator before production release) are supported in V1 but **never as the default**. They massively slow the pipeline and add logistics complexity; most low-mid tier jobs do not economically justify them.

**Eligibility rules — hard proof is allowed when at least one applies:**

- Print process is `OFFSET` or `FLEXO`
- Finishing includes specialty layers (`FOIL_GOLD`, `FOIL_SILVER`, `SPOT_UV`, `EMBOSS`)
- MOQ > 1000 units
- Order total ≥ $2000
- Regulated packaging (FDA-bearing label, dietary supplement, etc.)
- First production run with this partner for this creator
- Provider's PreflightProfile flags the spec as `requireHardProof = true`

**Pricing:**

- **Free hard proof** if order total ≥ $2000 OR creator is on the Master tier (per PLATFORM_SPEC.md)
- **$35–$75 add-on** otherwise (admin-configurable per-partner; default $50). Creator pays shipping in either case.
- **Provider-required hard proof** (i.e. provider's profile says required): creator still pays unless the order qualifies for free.
- **Repeat hard proofs after a Change Order**: always charged. See §1 Layer 5 / Change Order rules — material change post-hard-proof triggers a new hard-proof cycle that the creator pays for. This prevents iterative-hard-proof abuse.

**Operational requirements (mandatory for hard-proof orders):**

- Tracked delivery (UPS / FedEx with tracking). No untracked mail.
- Proof receipt confirmation by creator (auto-confirmed at carrier "Delivered" event, or creator clicks "I received the proof").
- Explicit approval deadline: **5 business days from receipt** → 7-day gentle pause → 14-day auto-cancel + refund of unprinted portion.
- Schema persists `hardProofVersion` (incremented each proof cycle) and `hardProofApprovedAt` for audit defensibility.

**Why this policy:** allowing free unlimited hard proofs creates massive operational abuse potential. Pricing it makes creators self-select for when they really need physical confirmation; the eligibility gates ensure high-stakes jobs aren't blocked from getting one.

### 5.6 PreflightProfile snapshot at submission (legal reproducibility)

**Rule:** when an order moves to `SUBMITTED`, the full `PreflightProfile` is **snapshotted** into the order's PreflightReport — not stored as a foreign-key reference. This is non-negotiable for legal defensibility.

```
PreflightReport {
  preflightProfileId: string                 // for analytics + grouping
  preflightProfileSnapshot: jsonb            // immutable copy of the entire profile at SUBMITTED time
  preflightProfileVersionAtSubmit: int       // for human-readable audit
  ...
}
```

**Why this matters:** profiles get updated. Tolerances change. Substrate options get added or removed. PitStop XML rules get patched. If we only persisted an FK, an order audited months later would be evaluated against rules that didn't exist at submission — making approvals legally ambiguous and disputes impossible to reproduce.

**Consequences:**
- In-flight orders use the snapshot they were evaluated against at submission. Profile updates do NOT mutate in-flight validation.
- New orders use whatever PreflightProfile version is `ACTIVE` at their submission time.
- The snapshot includes the PitStop XML rule set + the iLaunchify wrapper config + tolerances + supported finishes / substrates / processes.
- When an admin updates a PreflightProfile, the old version is preserved in `PreflightProfile.previousVersions[]` for analytics and legal lookup.

This pattern is industry standard in enterprise print procurement (Cimpress, HH Global): every job's audit trail is reproducible against the exact rules it was approved under.

**Change Order pricing rules:**

- Provider must re-quote within 1 business day SLA.
- Quote includes the price delta and a one-paragraph rationale.
- If creator approves → Stripe charges (or refunds, if delta is negative) the difference.
- If creator rejects → order continues at original spec/price.
- If provider initiates the Change Order due to their own error (e.g. they realized they quoted a substrate they don't carry) → they cannot charge more; only the creator-initiated changes can result in upcharge.

**Refund logic:**

- Pre-`PRODUCTION_LOCKED` cancellation: full refund.
- Post-`PRODUCTION_LOCKED` cancellation: depends on production stage. Pre-press materials cost (plates, dies) is non-refundable. We need a `cancellationPolicy` JSONB on each PartnerService that defines refund rules per state.
- Failed delivery (provider's fault — color drift, wrong size, etc.): full refund or re-do at provider cost. Platform ops adjudicates. Provider's account is debited via Stripe Connect transfer reversal.

## 6. SLA model

| Event | Acting party | SLA | Breach action |
|---|---|---|---|
| Preflight result | Automated | < 60 sec (95th percentile) | If queue backs up: page on-call SRE; show creator "preflight queued, sorry, ETA 15 min" |
| Provider Quick Review | Provider prepress tech | 4 business hours | Auto-escalate to Provider Admin user; +12h → reassign or refund |
| Provider Full Prepress (soft proof) | Provider prepress tech | 24 business hours | Notification to provider admin; +24h → escalate to platform ops; +72h → auto-refund and reassign |
| Provider Full Prepress (hard proof) | Provider prepress tech | 5 business days from request to mail-out | +2 days → escalate |
| Creator approval (Gate A) | Creator | 72 hour gentle nudge → 7 day pause | At 7 days: order pauses, proof expires after 30 days |
| Change Order quote | Provider | 1 business day | Escalate to provider admin |
| Change Order creator response | Creator | 3 business days | Pause; reminders at 1d, 3d, 7d |
| Production start (Gate B) | Provider production | 2 business days after `PRODUCTION_LOCKED` | Escalate |

**"Business hours" definition:** each Partner declares their working calendar (timezone + days + hours + holiday list). SLAs pause outside of business hours. We already have timezone fields on the Partner model; need to add a calendar/holiday model.

**Tracking:** Every SLA-bound event has a `slaDueAt` timestamp computed at state entry. A background worker scans for breaches every 5 minutes and triggers notifications / state transitions.

## 9. What happens OUTSIDE the platform

Even with the platform doing maximum work, providers operate physical equipment. The platform does not (and should not) manage:

- **Press calibration and color management** — providers maintain ISO 12647 compliance themselves. We can require them to attest to it during onboarding.
- **Physical proof shipping** — hard proofs go via UPS / FedEx outside our system. We track only the "proof requested" and "proof received by creator" timestamps.
- **Press operator training, machine maintenance, plate-making** — entirely provider's responsibility.
- **Substrate / consumable inventory** — provider's problem. If they can't fulfill, they reject the order (or we add a substrate-availability declaration to PreflightProfile so we don't quote unfulfillable orders in the first place).
- **Provider's internal QC** — they run their own end-of-line checks. We require evidence (photos) at `SHIPPED` for high-stakes orders.
- **Provider's relationship with paper / ink / die suppliers** — out of scope.
- **Provider's accounting / invoicing of their staff** — Stripe Connect handles payouts to the partner org; how they pay their staff is their problem.
- **Disposal of waste / sustainability certifications** — we can surface their certifications but don't manage them.
- **The actual ink hitting the paper** — obviously.

What the platform DOES manage end-to-end:

- File state, versioning, hashes
- All approvals, audit logs, liability transfers
- All money flow (Stripe Connect)
- All structured communication between parties
- All SLA tracking and escalations
- All compliance + preflight gates
- All shipping label generation (V2) and tracking (V1)
- All change orders and re-quotes

## 10. Provider Confidence Modes

Added Pavel 2026-05-25 — a cross-cutting concept the original spec was missing.

**The problem:** enterprise print houses and small digital shops operate with very different risk tolerances and operational philosophies. An offset packaging plant doing 50K-unit runs wants every file touched by a human; a small digital sticker shop printing 50 stickers wants maximum automation so the order ships same-day. Forcing both into a single operational model creates friction at both ends — the enterprise feels exposed, the digital shop feels strangled.

**The solution:** each Partner declares a **Confidence Mode** that sets sensible defaults across multiple subsystems simultaneously. Individual settings can still be overridden, but the mode is the lever that 95% of partners only ever touch.

```prisma
enum PartnerConfidenceMode {
  AUTOMATION_FIRST   // Lean digital shops, sticker/business-card volume, fast turnaround
  BALANCED           // Default for most general-purpose print + packaging providers
  WHITE_GLOVE        // Enterprise offset / specialty / premium — every file touched
}
```

### 10.1 What each mode tunes by default

| Setting | AUTOMATION_FIRST | BALANCED | WHITE_GLOVE |
|---|---|---|---|
| `autoReleaseMaxRiskScore` | 7 | 3 (global default) | 0 (touch every file) |
| Default review route for non-auto-released | QUICK_REVIEW | QUICK_REVIEW | FULL_PREPRESS |
| Hard proof eligibility | Specialty finishes only | Per §5.5 rules | Default for orders ≥ $1k |
| Revision-round hard cap | 3 | 3 | 5 (more room for refinement) |
| Provider Quick Review SLA | 2 business hours | 4 business hours | 8 business hours (deeper review acceptable) |
| Provider Full Prepress SLA | 12 business hours | 24 business hours | 48 business hours |
| Soft proof generation | Optional | Required for high-risk | Required for every order |
| Compliance scan strictness | Pass at WARN | Stop at WARN | Stop at WARN + AI-assisted review |
| Per-creator first-order treatment | Standard | First order gets +5 risk score | First order always FULL_PREPRESS |

### 10.2 How partners pick (and change) their mode

- During Phase 5 of partner onboarding (Integration layer per [[ilaunchify-partner-onboarding]]), partner picks one of the three modes with clear explainer copy.
- Admin can override (rarely — only when there's a misclassification dispute).
- Partner can change their own mode any time, BUT: changes only apply to **new orders submitted after the change** — no in-flight effect. (Preserves the snapshot/reproducibility rule.)
- Each change writes an AuditLog row + sends a courtesy email to platform ops (signal: who's tightening vs loosening).

### 10.3 Why this isn't just one slider

It's tempting to expose every setting individually and let partners tune. We resist:

- Most partners don't know what `autoReleaseMaxRiskScore = 7` means until they've run orders. A mode picker gives them a coherent starting posture.
- Three named modes match how providers actually describe themselves ("we're automation-first" / "we white-glove every job").
- A mode picker makes per-mode analytics legible: "AUTOMATION_FIRST partners have a 7% dispute rate vs WHITE_GLOVE 0.5%" is a useful insight; a 50-dimensional config matrix isn't.
- Custom partner profiles can still be built by starting from a mode and then overriding individual knobs.

### 10.4 What this DOES NOT change

- **Approval gates** still apply identically. Even AUTOMATION_FIRST partners must complete Gate B per order. Liability transfer is non-negotiable.
- **Change Order requirements** still apply identically. No mode allows skipping the re-quote flow.
- **Payment capture rule** still applies — all involved partners must approve before Stripe captures.
- **Audit logging** is identical across modes.

Confidence Modes only tune *defaults* and *thresholds*. The core safety architecture is the same for every partner.

## 12. Risks and mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Preflight engine (PitStop) has a bug; passes a bad file | High | Provider's Quick Review catches as last line of defense; track per-issue-code preflight miss rate; update profiles |
| Creator approves a proof with a typo | Medium | Liability transfer makes this their cost; UX nudges (spell check, double-tap "I confirm") reduce frequency |
| Provider goes offline mid-order | High | SLA breach → auto-escalate; in V1.5+, backup-provider reassignment with state migration |
| Color drift between proof and production | Medium | Delta-E tolerance per PreflightProfile; if breached, provider absorbs cost. Track per-partner drift rate; deactivate chronic offenders. |
| Creator and provider in a stalemate after 3 revision rounds | Medium | Forced escalation to platform ops; platform ops can refund, reassign, or close-no-fault |
| Provider declares a finishing capability they don't actually have | High | First production order with the capability triggers a hard-proof requirement and admin validation |
| Creator uses platform to design counterfeit / IP-infringing product | High | Pre-Gate-A: T&C clickthrough acknowledging IP responsibility; post-incident: takedown + ban; admin review queue for flagged designs |
| Compliance gate fails after preflight passes (label is print-OK but not FDA-OK) | High | Both gates must pass; order cannot advance to PROOF_PENDING_CREATOR until both PASS; explicit "compliance not yet passed" badge in wizard summary |
| Provider's PreflightProfile is too strict; orders never auto-release | Medium | Admin reviews provider's auto-release rate quarterly; warn if < 30% of eligible orders auto-release |
| Fraudulent creator requests refund after taking delivery | Medium | Require photographic evidence of defect; track creator dispute rate; require Master tier or successful order history for high-value orders |

All originally-open questions resolved by Pavel 2026-05-25. Underlying philosophy memorialized in [[ilaunchify-operational-philosophy-v1]] — operational trust over margin optimization for V1.

1. **Preflight engine — LOCKED: Enfocus PitStop Server for V1.** Behind an engine-agnostic `PreflightEngine` interface to avoid lock-in. Re-evaluate alternatives at 10k+ orders, when real error patterns are known. (See §1 Layer 2.)
2. **Provider Prepress Tech persona — LOCKED: separate role.** `PARTNER_ADMIN` / `PARTNER_PREPRESS` / `PARTNER_PRODUCTION` stay distinct in the schema even when small providers assign all three to one person. Per-technician analytics (rejection rates, revision efficiency, approval accuracy, SLA compliance) require the distinction. (See §2.)
3. **Auto-release default risk threshold — LOCKED: `≤3` at launch.** Phased rollout: Phase 1 (months 0-3) = ≤3 platform-wide to gather statistics safely. Phase 2 = expand to ≤5 only after low dispute / revision / override rates prove safety. Per-partner override via `PartnerService.autoReleaseMaxRiskScore` from day one. Provider Confidence Modes (§10) automate per-partner defaults. (See §1 Layer 3.)
4. **Hard cap on revision rounds — LOCKED: 3 with escalating friction.** Round 1 = standard. Round 2 = warning banner. Round 3 = red banner + "Final round before escalation" + Change Order shortcut. Track `provider.revisionRoundAverage` + `creator.revisionRoundAverage` as abuse / trust metrics. (See §4.)
5. **Hard proof shipping — LOCKED: supported in V1, NOT default.** Risk-triggered (offset / flexo / specialty finishes / MOQ > 1000 / order ≥ $2k / first run / regulated packaging / provider-required). Free if order ≥ $2000 or creator is Master tier. Otherwise $35-$75 add-on (creator pays shipping always). Tracked delivery mandatory. (See §5.5.)
6. **PreflightProfile authoring — LOCKED: JSON / structured config V1.** No UI builder yet. V1 investment goes into validation / versioning / diffing / rollback / test mode, NOT visual editing. Form-based editor lands V1.5 once real profile variance is understood. (See §1 Layer 2 + §7.1.)
7. **Creator-facing liability language at Gate A — LOCKED: engage specialized print/manufacturing/UCC counsel.** NOT generic startup counsel. Gate A/B language is core infrastructure, not UI copy. Must include click-through versioning, jurisdiction handling, proof-retention policy, arbitration/refund language, and Delta-E tolerance definitions. (See §1 Layer 4.)
8. **In-flight orders when a PreflightProfile updates — LOCKED: freeze profile version at SUBMITTED + persist full snapshot.** Snapshot includes the entire PitStop rule set + wrapper config + tolerances + supported finishes/substrates/processes — not just an FK. Profile updates never mutate in-flight validation. (See §5.6 + §7.3.)
9. **Hard proof revisions after shipment — LOCKED: treated as Change Order; creator pays.** Material changes (artwork / color / finish / substrate / dimensions) after a hard proof trigger a new hard-proof cycle that the creator funds. Provider may require this. Prevents iterative-hard-proof abuse. Schema persists `hardProofVersion` + `hardProofApprovedAt`. (See §5.5 + §7.14.)

**New concept added in resolution: Provider Confidence Modes** (§10) — `AUTOMATION_FIRST` / `BALANCED` / `WHITE_GLOVE`. Each partner declares a mode that tunes auto-release thresholds, proof requirements, revision strictness, and SLA expectations as a coherent unit. Prevents the platform from forcing enterprise print houses and small digital shops into the same operational model.
