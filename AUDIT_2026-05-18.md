# iLaunchify Audit ‚Äî May 18, 2026

**Scope:** Read-only audit of `/Users/soundstation/Documents/FOD` (copied to `iLaunchify/FOD-reference/` for inspection). Four parallel investigations: frontend, backend, microservices/data layer, compliance engine.

**Owner question:** Rebuild from scratch, or salvage and restructure?

**Short answer:** Rebuild ‚Äî but salvage four specific assets that took real work to produce. Rationale and effort sizing at the end.

---

## TL;DR

The project's *intent* is sound: a white-label platform doing recipe building, white-label storefronts, and multi-jurisdiction compliance for food / beverage / supplements / pet food / baby food. The *execution* shows a codebase that started as a Sequelize/PostgreSQL Node monolith, was migrated halfway to a Prisma/CockroachDB + FastAPI microservices architecture, then stalled. Roughly half of what's on disk is dead, dormant, or scaffolding. The half that's alive is concentrated in three pieces: a 27-model Prisma schema, a couple of Python services (PDS, calculation), and a fragile-but-shipping recipe-builder UI.

The compliance engine ‚Äî arguably the platform's defensible moat ‚Äî is **scaffolded but empty**: one US/FDA rule pack at 21 lines, zero rules for EU/FSANZ/CA, zero category-specific rules for supplements / baby food / pet food, and a single hardcoded sodium-plus-trans-fat threshold doing duty as "FDA compliance check."

## 1. Frontend (Next.js 15 + React 18)

**Size:** 793 source files in `frontend/src/` (422 .tsx + 371 .ts).

**The damning evidence:**
- **21 recipe-builder variants** in `frontend/src/components/product-builder/`. The "production" one ‚Äî `EnhancedRecipeBuilder.tsx` ‚Äî is **7,812 lines** in a single file. Siblings include `BulletproofRecipeBuilder`, `StableRecipeBuilder`, `SafeProductBuilder`, `ProperMUIRecipeBuilder`, `UnifiedRecipeBuilder`, `EnhancedRecipeBuilderRefactored`, `EnhancedRecipeBuilderCockroachDB`, `RecipeBuilder_temp`, `MinimalTest`, and `Phase2/3/4EnhancedProductBuilder`. The adjectives are the diagnosis.
- **Build silences errors.** `frontend/next.config.js` has `eslint.ignoreDuringBuilds: true` and `typescript.ignoreBuildErrors: true`. The 4.9 MB `tsconfig.tsbuildinfo` is committed, suggesting type-check was once run; build currently needs `NODE_OPTIONS='--max-old-space-size=8192'` to not OOM.
- **230 `localStorage` references across 52 files.** The audit-in-repo flagged 4 files ‚Äî actual scope is ~13x larger. SSOT principle is not enforced anywhere.
- **Four state libraries declared, three barely used.** `@reduxjs/toolkit` (4 files), `zustand` (1 file), `react-query` (0 imports ‚Äî declared, never used), Context API (7 files). Real state lives in Context + localStorage.
- **Storefront is mocks.** `app/products/page.tsx` imports `mockProducts`. `lib/cart.ts` exists with localStorage-backed `CartManager` ‚Äî **zero imports anywhere in the codebase**. No `/cart/`, `/checkout/`, or `/shop/` routes exist.
- **Vendored jQuery plugin** (`nutrition-label-jquery-plugin`) is checked into `app/nutrition-label-master/` as a routable app segment, alongside a literal `.zip` file of the same.
- Both `package-lock.json` (817 KB) and `yarn.lock` (449 KB) committed. `.env` and `.env.local` committed (likely with secrets ‚Äî rotate before sharing the repo).
- 49 `@fontsource/*` packages bundled.

**What's not duct tape:** the admin panel (`components/admin/`, 40 files) has real structure ‚Äî analytics, compliance, integrations, languages-markets, logistics, security, templates. The shape is non-trivial, though wiring vs. demo state is hard to judge without runtime testing.

**Verdict:** Rebuild advised.

## 2. Backend (Node/Express)

**Size:** 14,661 LOC in 58 top-level `.js` files; ~16,100 with subdirs.

**The live system is much smaller than the file count suggests.** `backend/index.js` mounts exactly 5 routers (`routes`, `auth-routes`, `vendor-dashboard-routes`, `creator-routes`, `admin-routes`, `marketplace-order-routes`). Twelve more routers (ingredient, label, vendor-profile, support, payment, message, template, product‚Ä¶) are declared in the file but **commented out** ‚Äî dormant, not deleted.

**Duplicate route files ‚Äî 10 dead variants:**

|---|---|
| `auth-routes.js` | `auth-old.js`, `auth-prisma.js`, `auth-routes-minimal.js`, `auth-routes-old-complex.js`, `auth-routes-prisma-complete.js` |
| `admin-routes.js` (158 LOC) | `admin-routes-old.js` (553), `admin-routes-prisma.js` (byte-identical to live) |

**Dual-write theatre.** `dualWrite.js` (54 LOC) is a generic helper that fires writes at a Python sidecar fire-and-forget ‚Äî *no* reconciliation, just shadow. Ten `*-dual-write.js` middlewares exist; **none are actually mounted in `index.js`** (every `app.use(...dualWrite, ...)` line is commented out). `discrepancies.json` holds 5 entries spanning a 44-minute window on 2025-08-31, every one with `pythonResult: null, error: ""`. The reconciliation pipeline ran once 8+ months ago and was abandoned.

**Three auth systems started, one alive.** Local JWT + Google OAuth (inlined in `auth-routes.js`) is the live path. Keycloak: setup script exists, zero references in code. Supabase: schema file exists, zero `supabase` references in backend `.js`. `google-auth-routes.js` and `google-oauth.js` exist standalone, never mounted.

**Kong is configured and bypassed.** `kong.yaml` defines 5 services and is brought up by docker-compose, but the frontend `env.unified` sets `NEXT_PUBLIC_API_URL=http://localhost:4000` ‚Äî directly to backend. Kong upstreams reference `mock-admin-api`/`mock-public-api` containers that don't exist in compose.

**~40% of `/backend/` is dead/dormant/migration scaffolding.** Files like `user-management-cutover-strategy.js`, `user-management-retirement-plan.js`, `user-management-shadow-traffic.js`, `post-migration-optimization.js`, `production-deployment.js`, `system-validation.js` add up to ~4,500 LOC of ceremony for a migration that never finished.

**What's worth keeping:** `services/orderLifecycleService.js` (a hand-crafted 17-state order FSM with `assertTransition` ‚Äî real business logic), `marketplace-order-routes.js`, the `services/{categoryService,partnerService,productService,userService,vendorService}.js` Prisma layer, and the `api/v1/` namespace which looks like an aborted clean rewrite attempt that deserves a closer look before deciding.

**Verdict:** Salvage ~2K LOC of real domain logic; rebuild around it.

## 3. Microservices & Data Layer

**Size:** ~31,300 LOC across `/services/`.

**The data layer is the strongest piece of the project.** `prisma/schema.prisma` has 27 production-shaped models ‚Äî Market, RulePack/Version, Template/Version/Assignment, Recipe/Ingredient/RecipeIngredient, Product, Order/OrderItem, Asset, AuditLog, SystemLog, etc. Datasource is `cockroachdb`. This is the asset most worth preserving. Caveat: exactly **one migration** lives in `migrations/` (`20250830022506_init`) ‚Äî schema drift since Aug 30 2025 is being handled by ad-hoc SQL (`migrate-database.sql`, `reorganize-database.sql`).

**The microservices fleet is half-built and not integrated.**

| Service | LOC | State |
|---|---|---|
| `services/pds` (Product Data Store) | 2,205 | Real, ambitious ‚Äî versioning, templates, MinIO assets, events |
| `services/calculation` | 484 | Real ‚Äî FastAPI calculation engine |
| `services/data-api` | 3,861 | **Seven competing `*-server.js` variants** in one folder |
| `services/python/user-management` | 5,100 | Real |
| `services/python/product-management` | 2,600 | **Duplicates `services/pds`** with no documented winner |
| `services/python/order-management` | 2,200 | **Duplicates `services/orders` (412 LOC)** |
| `services/python/food-safety` | 71 | Stub |
| `services/inventory/orders/payments/subscriptions` | 412‚Äì859 ea. | Real FastAPI+Prisma |

**Docker compose declares 28 services with 4 healthchecks.** Both PostgreSQL **and** CockroachDB are spun up ‚Äî directly contradicting the "CockroachDB as single source of truth" doctrine the audit report and architecture docs claim. Prometheus scrapes 2 targets; the 20+ microservices are invisible to monitoring.

**The migration to "unified database" was authored but never landed.** `migrate-to-unified-database.py` claims to strip SQLAlchemy/SQLModel/Alembic in favor of Prisma; root `requirements.txt` still pins `sqlmodel==0.0.24`, `psycopg2-binary`, `sqlalchemy-cockroachdb==2.0.3`. `start_nutrition_worker.py` imports `workers.nutrition_worker` from `./app` ‚Äî that path doesn't exist; the launcher is broken as-shipped.

**Supabase is a dead artifact.** Schema file + setup doc exist, zero call sites in code.

**Verdict:** Schema is genuinely well-designed (keep it). The "microservices platform" framing is currently aspirational.

## 4. Compliance / Regulatory Engine

**This is the section that should most affect the rebuild decision.**

The compliance moat is the platform's hardest-to-replicate value. The current state:

- **One rule pack file exists: `rule_packs/fda-us-2024.01.json` ‚Äî 21 lines.** Defines mandatory display field list, two rounding rules (`energy_kcal`, `fat_total_g`), two %DV entries. Duplicated byte-for-byte at `frontend/rule_packs/`. **No EU, no FSANZ, no CA, no UK, no supplements, no baby food, no pet food rule packs.**
- **Jurisdictions are a hardcoded mock in `JurisdictionContext.tsx`** ‚Äî US, EU, CA only, with a code comment: *"in production this would come from an API"*. The "rules" per jurisdiction are font-size and required-field stubs.
- **The real "compliance check" is one function:** `CalculationEngine.check_compliance()` in `services/calculation/calc_service.py:296-340`. It checks `totalSodium > 600` and `totalTransFat > 0.5` against hardcoded thresholds, and tags products organic by substring-matching `"organic"` in ingredient names. That's it. One sodium threshold for all products ‚Äî no category awareness.
- **Categories supplements / baby / pet are not even in the enum.** `ComplianceRule.category` in `ComplianceEngine.tsx` is `'food' | 'beverage' | 'non-food'`. No DSHEA logic, no AAFCO logic, no infant-formula heavy-metal limits.
- **USDA reference data is the substantial asset** ‚Äî Foundation, Branded, SR Legacy, Survey/FNDDS, chunked into ~89+ files under `frontend/public/data/usda/chunks/`. EFSA, Health Canada, FSANZ: zero data.
- **The nutrition label is rendered by a third-party jQuery plugin** (`nutrition-label-jquery-plugin@11`). The platform's "label generation" is USDA lookups feeding that plugin.

**Verdict on compliance:** Of the 5 categories the platform claims to serve, **0 have working compliance rules**. What ships is a USDA-backed FDA-format nutrition label generator with a single sodium check, plus a framework of empty rule-pack DB tables. This is the gap that most matters for the product's value proposition.

## What's worth keeping

These are the assets a rebuild should port forward rather than redo:

1. **`prisma/schema.prisma`** ‚Äî 27-model CockroachDB schema. The data model is the most production-grade artifact in the project.
2. **`services/calculation/calc_service.py`** ‚Äî 484-line nutrient summation + rule-pack rounding engine. Compact, isolated, real.
3. **`services/pds/`** ‚Äî 2,205-line Product Data Store with versioning and asset handling. Ambitious but real; worth porting carefully.
4. **`services/orderLifecycleService.js`** ‚Äî hand-crafted 17-state order FSM with explicit transition guards. Hard to redo.
5. **USDA data + chunking scheme** under `frontend/src/data/usda/` and `frontend/public/data/usda/chunks/`. The ingestion pipeline cost real time.
6. **`frontend/src/lib/nutrientCalculator.ts`** + **`lib/nutrition/label-generation.ts`** ‚Äî domain logic isolated from UI.
7. **TypeScript domain types** under `frontend/src/types/` ‚Äî recipe / product / market / compliance type definitions.
8. **`frontend/src/components/nutrition/NutritionFactsRenderer.tsx`** ‚Äî presentational FDA-style label rendering (replace the jQuery plugin with this).
9. **`kong.yaml`** as a *specification* of intended routing, not the running config.
10. **The `/docs/` folder's architecture documents** ‚Äî the *intent* in `COCKROACHDB_CENTERED_ARCHITECTURE.md`, `DATABASE_CONSOLIDATION_PLAN.md`, etc., is sound. Read these into the new project as design priors.

## What's not worth keeping

The 21 recipe-builder variants. The 10 dead route duplicates. The dual-write middlewares and discrepancyLogger. The user-management-cutover/retirement/shadow/monitoring files (~4,500 LOC). The Sequelize-era root `index.js` and `order-routes.js`. The vendored jQuery plugin + zip. The 7-variant data-api folder. The Supabase artifacts. Keycloak setup. `start-database.sh` (legacy PostgreSQL). `start_nutrition_worker.py` (broken import path). The 49 bundled fonts.

## Recommendation

**Rebuild ‚Äî surgically, not from a blank page.**

Start a fresh repo at the root of `iLaunchify/` with a deliberately small core:
1. **Monorepo (Turborepo or pnpm workspaces), not a microservice fleet on day one.** One Next.js app + one shared Prisma client + one Python service for calculation/compliance. Microservices later if scale demands.
2. **Port the Prisma schema verbatim** ‚Äî it's the strongest piece. Generate the initial migration cleanly.
3. **Port the calculation engine, order FSM, USDA pipeline, and PDS schema** as standalone modules with tests.
4. **Build the compliance engine intentionally**, treating it as the product's core IP rather than a checkbox. This is the work that wasn't done in the current codebase; it's the work most likely to make the rebuild worth it. Pick **one** category √ó one jurisdiction (e.g., US food labeling per 21 CFR 101) and ship it complete before adding the second.
5. **Pick one auth path (Supabase Auth or NextAuth + Postgres), one state strategy (TanStack Query for server state + Zustand for client state), one ORM (Prisma), one font system (3 fonts max).**
6. **Defer everything that isn't day-1 essential:** Kong, Keycloak, Storybook, the marketplace, multi-jurisdiction, Grafana dashboards, the white-label storefront. Add them when there's a working core to wrap them around.

**Rough effort sizing** (one experienced full-stack engineer, deliberate scope):
- Weeks 1‚Äì2: Repo skeleton, Prisma schema port, auth, basic CRUD for Product/Recipe/Ingredient, Stripe wiring.
- Weeks 3‚Äì5: Recipe builder UI (single, no variants) + USDA ingredient picker + calculation engine port.
- Weeks 6‚Äì8: US/FDA compliance rule pack ‚Äî *real* this time, with category awareness for supplements (DSHEA Supplement Facts panel) as a second tier.
- Weeks 9‚Äì10: Order lifecycle, payments, basic admin.
- Weeks 11‚Äì12: One white-label storefront template, hardening, deploy.

That's ~3 months to a *narrower but actually working* product. The current codebase couldn't reach the same milestone in 3 months because every change has to navigate 10K+ lines of migration scaffolding and 21 competing recipe-builder implementations.

If the rebuild is right, the FOD-reference folder becomes a read-only quarry ‚Äî extract the 10 listed assets, leave the rest.
